## 2026-06-20 - MCDEV source trace for real 26.1 item icons

Goal: replace the intentional 26.1 placeholder item icons by finding the vanilla caller/render path instead of guessing 26.2 snapshot APIs.

MCDEV versions available for this trace:
- `26.1`: decompiled + indexed + callgraph.
- `1.21.11`: decompiled + indexed + callgraph.
- No `26.2` snapshot source was available in MCDEV during this pass; comparison to 26.2 is from the local `fabric-26.2-dev` provider shape already in this repo.

Typical vanilla callers verified:
- 26.1 `AbstractContainerScreen.extractSlot(...)` calls `GuiGraphicsExtractor.item(...)` or `GuiGraphicsExtractor.fakeItem(...)`, then `itemDecorations(...)`.
- 26.1 `Gui.extractSlot(...)` calls `graphics.item(player, itemStack, x, y, seed)` for the hotbar/HUD path.
- 26.1 `ItemDisplayWidget.extractWidgetRenderState(...)` calls `graphics.item(...)` for simple item widgets.
- 1.21.11 has the same conceptual callers under the old render naming: `AbstractContainerScreen.renderSlot(...)`, `Gui.renderSlot(...)`, and `ItemDisplayWidget.renderWidget(...)` call `GuiGraphics.renderItem(...)` / `renderFakeItem(...)`.

26.1 item icon pipeline verified through MCDEV:
1. `GuiGraphicsExtractor.item(...)` resolves the stack into a `TrackingItemStackRenderState` via `Minecraft.getItemModelResolver().updateForTopItem(...)`.
2. It adds a `GuiItemRenderState` into `net.minecraft.client.renderer.state.gui.GuiRenderState.addItem(...)`.
3. `GuiRenderer.prepareItemElements()` gathers model identities, creates/reuses `GuiItemAtlas`, and calls `GuiItemAtlas.getOrUpdate(TrackingItemStackRenderState)` for each non-oversized item.
4. `GuiItemAtlas.getOrUpdate(...)` is public in exact 26.1. It allocates a slot and calls private `drawToSlot(...)` when the slot is empty/stale.
5. `GuiItemAtlas.drawToSlot(...)` performs the real vanilla icon render into an offscreen atlas texture:
   - sets `RenderSystem.outputColorTextureOverride` and `RenderSystem.outputDepthTextureOverride`,
   - sets an orthographic projection with `ProjectionMatrixBuffer`,
   - chooses `Lighting.Entry.ITEMS_3D` or `ITEMS_FLAT` from `item.usesBlockLight()`,
   - calls `item.submit(poseStack, submitNodeCollector, 15728880, OverlayTexture.NO_OVERLAY, 0)`,
   - calls `featureRenderDispatcher.renderAllFeatures()`,
   - flushes `bufferSource.endBatch()`.

Important exact-26.1 API availability:
- `GameRenderer.getSubmitNodeStorage()` exists and returns `SubmitNodeStorage`, which implements `SubmitNodeCollector`.
- `GameRenderer.getFeatureRenderDispatcher()` exists.
- `GameRenderer.getLighting()` exists.
- `FeatureRenderDispatcher.renderAllFeatures()` exists with no arguments and uses its own private `bufferSource` / `crumblingBufferSource`.
- `GuiItemAtlas` exists with public constructor and public `getOrUpdate(...)`.
- `GuiItemAtlas.SlotView.textureView()` returns `GpuTextureView`; `GpuTextureView.texture()` is public, so slot readback can obtain the backing `GpuTexture` without reflecting the atlas texture field.
- `CommandEncoder.copyTextureToBuffer(GpuTexture, GpuBuffer, long, Runnable, int, int, int, int, int)` exists for region copy.
- `CommandEncoder.mapBuffer(...)` is public in exact 26.1.
- `MultiBufferSource.immediate(ByteBufferBuilder)` exists, but using a separate buffer source with the game `FeatureRenderDispatcher` would not flush the same internal buffer used by `renderAllFeatures()`.

Comparison with 1.21.11:
- 1.21.11 uses `GuiGraphics.renderItem(...)` naming; 26.1 uses `GuiGraphicsExtractor.item(...)` naming.
- 1.21.11 keeps the atlas implementation inside `GuiRenderer` as `renderItemToAtlas(...)`, `submitBlitFromItemAtlas(...)`, and atlas coordinate fields.
- 26.1 factors that atlas code into `GuiItemAtlas`, which is better for us because `getOrUpdate(...)` is public and can be reused instead of copying private `drawToSlot(...)`.
- The model resolver signature `ItemModelResolver.updateForTopItem(ItemStackRenderState, ItemStack, ItemDisplayContext, Level, ItemOwner, int)` is the same in 1.21.11 and 26.1.

Comparison with local 26.2 snapshot provider shape:
- The local 26.2-dev route relies on snapshot-style APIs such as `GpuFormat`, short accessor names like `gameRenderer.lighting()`, and a different feature-dispatch shape. Those should not be imported into exact 26.1.
- Exact 26.1 wants `TextureFormat`, `gameRenderer.getLighting()`, `gameRenderer.getFeatureRenderDispatcher()`, and no-arg `FeatureRenderDispatcher.renderAllFeatures()`.

Recommended implementation route for 5.5:
- Replace the placeholder branch in `Minecraft261ItemTextureProvider` with a vanilla-atlas path:
  1. resolve the `ItemStack` into `TrackingItemStackRenderState` using `ItemModelResolver.updateForTopItem(...)`,
  2. obtain `FeatureRenderDispatcher` from `mc.gameRenderer.getFeatureRenderDispatcher()`,
  3. obtain matching `SubmitNodeStorage` via `featureDispatcher.getSubmitNodeStorage()` or `mc.gameRenderer.getSubmitNodeStorage()`,
  4. reflect the `FeatureRenderDispatcher.bufferSource` field, because `renderAllFeatures()` renders into that internal buffer and `GuiItemAtlas` must flush the same buffer with `endBatch()`,
  5. instantiate/reuse a `GuiItemAtlas(submitNodeStorage, featureDispatcher, reflectedBufferSource, textureSize, slotTextureSize)`,
  6. call `getOrUpdate(renderState)`,

- A one-slot `GuiItemAtlas` with `textureSize == slotTextureSize == 32` appears plausible because the constructor does not enforce the vanilla 512 minimum, but vanilla `computeTextureSizeFor(...)` clamps to 512. - Reusing the game `FeatureRenderDispatcher` means `renderAllFeatures()` clears submit nodes globally. This should run only on the render/game thread and should avoid interleaving with a normal frame render. Existing provider code should keep the same game-thread scheduling discipline.
- Oversized GUI items are handled separately by vanilla `OversizedItemRenderer`; the first bridge version can ignore oversized handling if the returned icon endpoint only needs standard item icons. Record this as a known limitation if not implemented.

- `GuiItemAtlas.drawToSlot(...)` is the key reusable recipe: set output color/depth texture overrides, install an orthographic projection, scissor to the slot, choose `Lighting.Entry.ITEMS_3D` or `ITEMS_FLAT`, call `ItemStackRenderState.submit(..., LightCoordsUtil/FULL_BRIGHT equivalent, OverlayTexture.NO_OVERLAY, 0)`, then `FeatureRenderDispatcher.renderAllFeatures()` and `bufferSource.endBatch()`.
- Directly using `GuiItemAtlas` is unattractive for extraction: `getOrUpdate(...)` is package-private, `drawToSlot(...)` is private, and the atlas texture is meant for later GUI blits rather than readback. The practical bridge route remains mirroring `drawToSlot(...)` into our own copy-readable texture.
- In `1.21.11`, the analogous caller is `net.minecraft.client.gui.GuiGraphics.renderItem(...)`; it also creates `TrackingItemStackRenderState` and calls `ItemModelResolver.updateForTopItem(...)`, but the atlas code is inline inside `GuiRenderer.prepareItemElements()`/`renderItemToAtlas(...)` rather than extracted into `GuiItemAtlas`.
- Version comparison: conceptually stable path is `ItemModelResolver.updateForTopItem -> TrackingItemStackRenderState -> ItemStackRenderState.submit -> FeatureRenderDispatcher.renderAllFeatures -> BufferSource.endBatch`; exact class/method packaging differs between `1.21.11` and `26.1`.

Implication for 26.1 implementation:

- Prefer the exact-26.1 vanilla recipe around `GuiItemAtlas.drawToSlot(...)` as the source of truth.
- Avoid depending on `GuiItemAtlas` itself unless we deliberately add mixin/accessor plumbing and solve readback separately.
- Keep the bridge-owned offscreen texture approach because it gives us the missing `COPY_SRC`/readback capability while staying aligned with vanilla's icon submission pipeline.

## 2026-06-20 - MCDEV item icon caller/API sweep

User asked to find alternative exact-26.1 APIs for item icons by tracing typical vanilla callers instead of guessing from old item renderer APIs.

MCDEV versions available for this sweep: `1.19`, `1.21.11`, exact `26.1`.

Exact 26.1 findings:
- `net.minecraft.client.gui.render.GuiItemAtlas` exists and is the normal GUI item icon atlas owner.
- `GuiItemAtlas#getOrUpdate(TrackingItemStackRenderState)` is package-private and has exactly one caller: `GuiRenderer.prepareItemElements`.
- `GuiItemAtlas#drawToSlot(int,int,boolean,ItemStackRenderState)` is private and performs the real offscreen render: set output color/depth texture overrides, ortho projection, scissor, choose `Lighting.Entry.ITEMS_3D` vs `ITEMS_FLAT` from `item.usesBlockLight()`, call `item.submit(...)`, `featureRenderDispatcher.renderAllFeatures()`, and `bufferSource.endBatch()`.
- `GuiRenderer.prepareItemElements` is private and obtains/prepares the atlas through private `prepareItemAtlas`; the atlas slot is then only blitted back into GUI render state via private `submitBlitFromItemAtlas`.
- Typical inventory caller is `AbstractContainerScreen.extractSlot`: it computes `seed = slot.x + slot.y * imageWidth`, calls `graphics.fakeItem(...)` for fake slots or `graphics.item(...)` for normal slots, then `graphics.itemDecorations(...)`.
- `GuiGraphicsExtractor.item(...)` creates `TrackingItemStackRenderState`, calls `minecraft.getItemModelResolver().updateForTopItem(itemStackRenderState, itemStack, ItemDisplayContext.GUI, level, owner, seed)`, then adds a `GuiItemRenderState`.
- `ItemModelResolver#updateForTopItem(...)` has public callers from GUI, in-hand rendering, activation overlay, particles, block entity render-state extraction, and smithing preview. This is the stable model-resolution seam.
- `ItemInHandRenderer.renderItem(...)` confirms the smaller non-GUI recipe: create `ItemStackRenderState`, call `itemModelResolver.updateForTopItem(...)`, then `renderState.submit(...)`.
- There is no old-style `net.minecraft.client.renderer.ItemRenderer` / `Minecraft#getItemRenderer()` high-level icon renderer in exact 26.1.

1.21.11 comparison:
- No `GuiItemAtlas` class exists.
- `GuiGraphics.renderItem(...)` performs the same high-level extraction recipe: create `TrackingItemStackRenderState`, call `minecraft.getItemModelResolver().updateForTopItem(..., ItemDisplayContext.GUI, ...)`, submit `GuiItemRenderState`.
- `GuiRenderer.prepareItemElements` owns the atlas fields directly in 1.21.11; its private `renderItemToAtlas(...)` is structurally the ancestor of 26.1 `GuiItemAtlas#drawToSlot(...)`.
- 1.21.11 still has `Minecraft#getItemRenderer()` and `net.minecraft.client.renderer.entity.ItemRenderer`, but that class is low-level quad/glint submission and is absent in exact 26.1.

Conclusion:
- No better public exact-26.1 icon-to-pixels API was found in the typical caller path.
- The least brittle exact-26.1 bridge path is to reuse vanilla model resolution (`Minecraft#getItemModelResolver()` + `ItemModelResolver#updateForTopItem(...)`) and the verified render-state submission recipe, but own our offscreen texture/readback instead of reaching into private/package-private `GuiRenderer`/`GuiItemAtlas` internals.
- Avoid porting 1.21.11 code that depends on `Minecraft#getItemRenderer()` or `net.minecraft.client.renderer.entity.ItemRenderer`; that API boundary does not exist in exact 26.1.

## 2026-06-20 - MCDEV source trace: vanilla GUI item icon path in 26.1 vs 1.21.11

User request: find a typical caller for item icons / item stack rendering and look for alternative APIs that work on exact Minecraft 26.1. Source verification was done through MCDEV static source tools for exact `26.1` and `1.21.11`; no runtime test/build/deploy was performed in this pass.

### Exact 26.1 vanilla caller chain

MCDEV refs show `GuiGraphicsExtractor.item(...)` is the ordinary GUI item-icon entry point. Callers include hotbar/slots/inventory screens, recipe buttons, bundle tooltip slots, creative tab buttons, beacon items, stonecutter recipes, and `ItemDisplayWidget`.

The 26.1 callees for `GuiGraphicsExtractor.item(...)` show this sequence:

1. `GuiGraphicsExtractor.item(...)`
2. allocate `TrackingItemStackRenderState`
3. `Minecraft.getItemModelResolver()`
4. `ItemModelResolver.updateForTopItem(output, stack, ItemDisplayContext.GUI, level, owner, seed)`
5. create `GuiItemRenderState`
6. `GuiRenderState.addItem(...)`
7. later, `GuiRenderer.prepare()` calls private `prepareItemElements()`
8. `prepareItemElements()` uses `GuiItemAtlas.getOrUpdate(itemState.itemStackRenderState())`
9. `GuiItemAtlas.drawToSlot(...)` renders the item into the atlas texture using output color/depth texture overrides, ortho projection, `item.submit(...)`, `featureRenderDispatcher.renderAllFeatures()`, and `bufferSource.endBatch()`
10. `GuiRenderer.submitBlitFromItemAtlas(...)` submits a GUI blit from the atlas slot.

### 26.1 access / reuse caveat

`GuiGraphicsExtractor.item(...)` is public, but it only enqueues into a `GuiRenderState`; it does not synchronously return pixels or an image. The pixel-producing work is downstream in `GuiRenderer.prepareItemElements()` and `GuiItemAtlas.drawToSlot(...)`.

`GuiRenderer.prepareItemElements()`, `GuiRenderer.prepareItemAtlas(...)`, and `GuiRenderer.submitBlitFromItemAtlas(...)` are private. `GuiItemAtlas.drawToSlot(...)` is private. `GuiItemAtlas.getOrUpdate(...)` is callable inside the `net.minecraft.client.gui.render` package but the useful readback still requires owning/reading the texture, and vanilla's atlas texture is not exposed as a PNG/readback API.

Therefore, the most robust bridge approach remains: reuse the public model-state API (`TrackingItemStackRenderState` + `ItemModelResolver.updateForTopItem`) and mirror the vanilla atlas render recipe into a bridge-owned, copy-readable offscreen texture. Avoid trying to drive the live GUI render state or private vanilla atlas internals for PNG extraction.

### 1.21.11 comparison

`1.21.11` uses `GuiGraphics.renderItem(...)` as the equivalent public GUI entry point. MCDEV callees show the same high-level chain: allocate `TrackingItemStackRenderState`, call `Minecraft.getItemModelResolver().updateForTopItem(...)`, create `GuiItemRenderState`, then submit it to `GuiRenderState.submitItem(...)`.

The renderer internals differ: `1.21.11` has no separate `GuiItemAtlas` class. `GuiRenderer.prepareItemElements()` owns atlas fields/methods directly and calls private `renderItemToAtlas(...)`. Exact `26.1` refactors that atlas responsibility into `GuiItemAtlas`, but still keeps the pixel-producing draw/readback path private/non-returning.

### Alternative APIs considered

- Public GUI enqueue path: `GuiGraphicsExtractor.item(...)`. Useful for drawing into vanilla GUI, not suitable for bridge PNG extraction because it returns no image and depends on the active GUI frame/render state.
- Vanilla atlas path: `GuiRenderer.prepareItemElements()` / `GuiItemAtlas`. Matches the desired pixels, but key parts are private and the atlas is not a readback API.
- Lower-level public model-state path: `TrackingItemStackRenderState` + `ItemModelResolver.updateForTopItem(...)` + `ItemStackRenderState.submit(...)`. This is the best cross-version seam and is the path the 26.1 provider should keep using.
- Legacy `ItemRenderer`: present as `net.minecraft.client.renderer.entity.ItemRenderer` in 1.21.11, but exact 26.1 GUI icons are no longer centered around a public item renderer object. Do not try to port the old 1.21.11 shortcut by name.

## 2026-06-20 - MCDEV source trace: 26.1 item icon alternatives

Goal for this pass: find a vanilla caller/path for GUI item icons and identify 26.1-safe APIs instead of guessing from the bridge provider alone.

MCDEV verified exact 26.1 source path:
- Typical callers of `GuiGraphicsExtractor.item(...)` include `Gui.extractSlot(...)` for hotbar/offhand-style slots, `AbstractContainerScreen.extractSlot(...)` for inventory/container slots, `CreativeModeInventoryScreen.extractTabButton(...)`, `RecipeButton.extractWidgetRenderState(...)`, `OverlayRecipeComponent$OverlayRecipeButton.extractWidgetRenderState(...)`, `ClientBundleTooltip.extractSlot(...)`, `ItemDisplayWidget.extractWidgetRenderState(...)`, and beacon/stonecutter screens.
- `AbstractContainerScreen.extractSlot(...)` computes the slot stack/seed, calls `graphics.fakeItem(...)` for fake slots or `graphics.item(...)` otherwise, then calls `graphics.itemDecorations(...)`.
- `Gui.extractSlot(...)` handles the pop animation with `graphics.pose()`, calls `graphics.item(player, itemStack, x, y, seed)`, then `graphics.itemDecorations(...)`.
- `GuiGraphicsExtractor.item(...)` constructs `TrackingItemStackRenderState`, calls `Minecraft.getItemModelResolver().updateForTopItem(renderState, stack, ItemDisplayContext.GUI, level, owner, seed)`, and enqueues a `GuiItemRenderState` into `GuiRenderState.addItem(...)`.
- `GuiRenderer.prepare()` calls private `prepareItemElements()`.
- In 26.1, `GuiRenderer.prepareItemElements()` delegates item atlas management to `GuiItemAtlas`: it gets GUI scale, calls `prepareItemAtlas(...)`, and for normal-size items calls `GuiItemAtlas.getOrUpdate(itemState.itemStackRenderState())` before blitting the returned `SlotView`.
- `GuiItemAtlas.getOrUpdate(...)` allocates/cache-checks by `TrackingItemStackRenderState.getModelIdentity()` and calls private `drawToSlot(...)` for empty/stale slots.
- `GuiItemAtlas.drawToSlot(...)` is the canonical render body: set the target color/depth texture overrides, set an orthographic projection, enable scissor, choose `Lighting.Entry.ITEMS_3D` or `ITEMS_FLAT` from `item.usesBlockLight()`, call `item.submit(poseStack, submitNodeCollector, 15728880, OverlayTexture.NO_OVERLAY, 0)`, call `FeatureRenderDispatcher.renderAllFeatures()`, end the buffer batch, clear overrides, and pop pose.

MCDEV verified exact 1.21.11 comparison:
- The public-ish frontend name is `GuiGraphics.renderItem(...)` instead of `GuiGraphicsExtractor.item(...)`, but callers are the same conceptual set: `Gui.renderSlot(...)`, `AbstractContainerScreen.renderSlot(...)`, recipe buttons, bundle tooltip, creative tab, beacon, stonecutter, etc.
- `GuiGraphics.renderItem(...)` still constructs `TrackingItemStackRenderState` and calls the same `ItemModelResolver.updateForTopItem(...)` signature.
- `GuiRenderer.prepareItemElements()` in 1.21.11 owns the atlas logic directly and calls private `renderItemToAtlas(...)`; 26.1 splits that into `GuiItemAtlas.getOrUpdate(...)` / private `drawToSlot(...)`.
- `ItemStackRenderState.submit(PoseStack, SubmitNodeCollector, int lightCoords, int overlayCoords, int outlineColor)` has the same signature in 1.21.11 and 26.1.

26.1 support APIs verified by MCDEV:
- `Minecraft.getItemModelResolver()` exists.
- `ItemModelResolver.updateForTopItem(...)` exists with `ItemStackRenderState`, `ItemStack`, `ItemDisplayContext`, `Level`, `ItemOwner`, `int seed`.
- `TrackingItemStackRenderState` extends `ItemStackRenderState` and exposes `getModelIdentity()`.
- `FeatureRenderDispatcher.getSubmitNodeStorage()` and `renderAllFeatures()` exist.
- `SubmitNodeStorage` implements `SubmitNodeCollector`.
- `RenderSystem.outputColorTextureOverride` and `RenderSystem.outputDepthTextureOverride` exist in full 26.1 source, even though the summary index did not list them.
- `CommandEncoder.copyTextureToBuffer(...)` exists with region overload, for readback after rendering to an owned texture.

Conclusion / implementation seam:
- The viable 26.1 bridge strategy is not to call a public "render icon to PNG" API; vanilla does not expose one.
- The stable seam is to reproduce `GuiItemAtlas.drawToSlot(...)` into a bridge-owned texture, using the public model-resolution and submit APIs, then read the bridge-owned texture back.
- The current `Minecraft261ItemTextureProvider` is already aligned with the stable part of this seam: `TrackingItemStackRenderState`, `Minecraft.getItemModelResolver().updateForTopItem(...)`, `FeatureRenderDispatcher.getSubmitNodeStorage()`, `ItemStackRenderState.submit(...)`, and GPU texture readback.

Difficult / risky points to keep explicit:
- `GuiItemAtlas.drawToSlot(...)` and `GuiRenderer.prepareItemElements()` are private, so a direct call/mixin into them would be brittle. Prefer copying the small render body unless runtime behavior proves mismatch.
- `FeatureRenderDispatcher` exposes `getSubmitNodeStorage()` but not `bufferSource`; accessing the buffer source remains the intentional accessor/mixin caveat.
- 26.1 moved atlas ownership into `GuiItemAtlas`, so a 1.21.11 reflection strategy against `GuiRenderer` internals is not a clean port.
- Lighting must follow vanilla's per-item choice: `item.usesBlockLight() ? ITEMS_3D : ITEMS_FLAT`.
- RenderSystem output overrides/projection/scissor must be restored reliably on failure, because this code runs on the render/client thread.
- Runtime validation still pending: build, Prism install, WebSocket smoke, and actual texture PNG comparison have not been run.
